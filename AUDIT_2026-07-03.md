## P0 — релиз-блокер

### P0-1 — GUI/TUI/settings показывают один сейф, а destructive CLI работает с ДРУГИМ `[verified]`
**Самый опасный разрыв. Источник B; A/GUI-агент ошибочно счёл закрытым — перепроверено, B прав.**

- `securetrash` **жёстко хардкодит** контейнер и mount и **никогда не читает** `ST_VAULT_PATH`/
  `ST_VAULT_VOLUME`:
  - `securetrash/securetrash:278-280` — `VAULT_PATH="${HOME}/SecureVault.sparsebundle"`,
    `VAULT_VOLUME="/Volumes/SecretVault"`; grep по `securetrash/securetrash` на `ST_VAULT_*` → **пусто**.
- А лаунчер и GUI env ЧИТАЮТ — но только для СТАТУСА/дашборда:
  - `paranoid:16,19` — `VAULT_VOLUME="${ST_VAULT_VOLUME:-…}"`, `VAULT_CONTAINER="${ST_VAULT_PATH:-…}"`.
  - `gui/macos/ParanoidBar.swift:14-18,48-51,249-270` — хранит `vaultVolume`, показывает статус по
    нему, префиксует Terminal-команды `ST_VAULT_VOLUME=<…>`.
- **Итог:** пользователь в settings задаёт кастомный vault, GUI/TUI показывают его как «open», а
  `securetrash vault reset/destroy/open` молча оперируют дефолтным `~/SecureVault.sparsebundle` →
  **destructive-операция над не тем объектом, который видит пользователь.** «Bug #1»-фикс (экспорт
  `ST_VAULT_VOLUME` из GUI) — косметический, т.к. securetrash его игнорирует. Settings-панель
  (s16) с override делает вид, что работает.
- **Фикс:** либо (a) внедрить чтение `ST_VAULT_PATH`/`ST_VAULT_VOLUME` в `securetrash` (bash+ps1),
  либо (b) убрать `Vault volume`-override из GUI/TUI до реальной поддержки. Тест: при кастомных env
  `status/open/reset/destroy` бьют по одному контейнеру/mount. В destructive-confirmation показывать
  и container path, и mount path.

## P1 — data-loss / security / supply-chain

### P1-1 — Windows vaultwatch `--force` = не реальный safety-gate → форс-dismount открытых файлов `[verified]`
**Оба аудита независимо (A-security H1, A-process MED-3, B-P1).**
- `vaultwatch/windows/vaultwatch.ps1:257-258` — `Test-VwMountBusy` всегда `return $false`.
- `:260-265` — `Invoke-VwDismount` зовёт `Lock-BitLocker -ForceDismount` даже на не-force пути.
- `:470-478` — TTL non-force ветка → `-Force $false`, но внутри всё равно `-ForceDismount`.
- `windows/README.md:37,140-146` обещает «`--force` needed to dismount with open files» — busy не
  детектится вовсе. bash-версия делает реальный `lsof` (`vaultwatch/vaultwatch:665-672`).
- **Риск:** TTL истекает при открытом файле → форс-dismount → повреждение данных.
- **Фикс:** без `--force` не звать `-ForceDismount`; при неизвестном busy — fail-closed (warn +
  оставить сессию); реальный busy-детект (`Get-SmbOpenFile`/handle-скан). Pester: `Invoke-VwDismount
  -Force $false` не передаёт `-ForceDismount`. + реальный BitLocker-тест на Windows.

### P1-2 — Windows-инсталляторы hash-only, без проверки подлинности релиза `[verified]`
**Источник B-P1 (A покрыл только bash-деградацию — это отдельная, более серьёзная дыра).**
- `securetrash/windows/install.ps1:63-88` — качает `securetrash.ps1` + `SHA256SUMS`, сверяет
  `Get-FileHash`; проверки `SHA256SUMS.sig` **нет** (grep: ни `ssh-keygen`, ни `.sig`). SHA256SUMS
  идёт из того же release-канала → против подменённого релиза = ноль authenticity.
- Та же модель у всех: `vaultwatch/panic/ghostdraft/seedsplit /windows/install.ps1`.
- **Фикс:** Ed25519-verify через `ssh-keygen -Y verify` (паритет с bash); fail-closed при битой/
  отсутствующей подписи; hash-only только за явным `PT_ALLOW_HASH_ONLY=1`. Pester: valid/invalid/
  missing sig, missing verifier.

### P1-3 — Единый Ed25519-ключ подписи на все 5 репо = один blast radius `[verified]`
**Источник A-security H3.** Один pubkey вкомпилен в `install.sh:40`, `verify-releases.sh:12` и каждый
тул-инсталлятор; приватный ключ — один Actions-secret, реплицированный в 5 `release.yml:31`.
Компрометация Actions-secret любого репо ИЛИ вредоносный PR в любой `release.yml` → ключ, подписывающий
всю экосистему. Нет per-tool ключа, нет ротации, нет revocation (pubkey вкомпилен).
- **Фикс (архитектурный, отдельно):** per-repo ключи ИЛИ OIDC/hardware-signer + задокументированный
  путь ротации/отзыва.

### P1-4 — macOS/root-инсталляторы деградируют до hash-only без `ssh-keygen`; молчаливо `[verified]`
**Источник A-security M3/M4 + B-P1.** `install.sh:112-118` (+ standalone `*/install.sh:55-67`) — без
`ssh-keygen` только warn + продолжает по SHA256. Доки обещают «Nothing executes until verified»
(`README.md:62-66`). Спасает сейчас Darwin-gate (macOS ships openssh) — латентный HIGH.
Вдобавок вложенный инсталлятор глушит stderr провала: `install.sh:130` `>/dev/null 2>&1` → tampering
неотличимо от сетевой ошибки.
- **Фикс:** если pubkey задан и `.sig` ожидается — отсутствие verifier = hard-fail (opt-in
  `ALLOW_UNSIGNED_LEGACY=1`); пробрасывать stderr тула при провале; README уточнить, что «verified».

### P1-5 — seedsplit: passphrase спиллит на диск через here-string `<<<` `[verified]`
**Источник A-crypto M1.** `seedsplit/seedsplit:319,511` — `openssl … -pass fd:3 3<<<"$pp"`. Bash (вкл.
3.2) реализует `<<<` через temp-файл в `$TMPDIR` (600, тут же unlink, но inode восстановим). Частично
убивает fd:3-дизайн. Без FileVault — carving восстанавливает master-passphrase.
- **Фикс:** подавать пароль через pipe/process-substitution/`printf >&3` coproc.

### P1-6 — «crypto-shred» = `rm -rfP` sparsebundle, не уничтожение ключа `[verified]`
**Источник A-vault M2.** `securetrash:625-636` (reset), `:555-569` (destroy). На SSD/APFS удалённые
band/token-блоки физически выживают (wear-leveling + COW). Настоящий crypto-shred затирал бы token
→ шифртекст мгновенно нечитаем. **Честно disclosed** в result-строках (`:185-186,204-205`), но лейбл
«crypto-shred» щедрее механики. Слабый пароль + carving после reset → старый vault восстановим.
- **Фикс:** реально затирать token ИЛИ переименовать лейбл в честное «delete + recreate (recovery
  depends on passphrase)».

## Что сделано ПРАВИЛЬНО (сохранить при рефакторах)

- **Крипто реальное:** корректный leak-free Shamir GF(256) (`t≥2` guard, Лагранж, CSPRNG); реальные
  бэкенды (hdiutil AES-256 stdinpass, BitLocker Aes256 + SecureString); пароли/секреты **никогда не
  через argv** (stdin/SecureString/fd:3).
- **Fail-closed destroy:** tri-state `mounted/unmounted/unknown`, `unknown` отказывается удалять +
  TOCTOU-перепроверка; detach строго по dev-node; сильные shred-guard'ы (канонизация + denylist
  OS-корней и `/Volumes/<mount>`, reparse-point refusal, diskpart-injection hardening).
- **Честное самодокументирование физ-пределов** (SSD/COW/swap/clipboard/VSS/timing) — низкий
  snake-oil-фактор. VeraCrypt на Windows намеренно отключён (argv-leak) — anti-snake-oil.
- **panic:** s16-ложь честно исправлена, fail-visible fallback, реальный Windows P/Invoke (не заглушка).
